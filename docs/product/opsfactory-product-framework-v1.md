# OpsFactory 产品框架 v1

## 文档目的

本文档用于定义 OpsFactory 在较长周期内应保持稳定的产品框架，用来指导产品、设计、架构与交付协同。

OpsFactory 不应被定义为“运维场景里的通用大模型助手”，而应被定义为：

**一个以 AI Native 为底座、以人与智能体协同为运行方式、将 ITOM、ITSM 与 Agent 执行能力打通的一体化运维与服务运营平台。**

它当下的任务，是帮助少量高水平的人类专家管理并放大一支不断增长的智能体队伍。  
它未来的形态，是一个开放的 cyber factory：在这个平台里，智能体、能力、数据、流程与治理被组织成一个真正可运营的生产系统。

## 产品命题

OpsFactory 之所以成立，是因为传统运维产品与通用 AI Agent 都不够完整。

传统 ITOM 与 ITSM 系统擅长记录、展示、流程管理与治理，但不擅长推理、归因、总结、建议与执行。  
通用 AI Agent 擅长对话与任务执行，但缺少稳定的运维上下文、服务管理结构、责任边界与生产级管理能力。

OpsFactory 要解决的，不是“再做一个会聊天的产品”，而是把这两类能力打通：

- 让 ITOM / ITSM 数据成为真实上下文
- 让智能体承担分析、诊断、生成与执行工作
- 让平台把洞察、决策、行动与能力沉淀串成闭环

OpsFactory 的本质，不是“回答问题”，而是“组织智能体成为可管理的生产力”。

## 产品定义

OpsFactory 是一个面向运维与服务运营的人机协同平台。

它由四个部分共同构成：

1. 来自 ITOM 的运行上下文
2. 来自 ITSM 的服务与流程上下文
3. 由 Agent 承担的分析、诊断、生成与执行能力
4. 面向第三方与组织内部沉淀的开放能力层

因此，OpsFactory 既不是一个纯聊天产品，也不是一个纯看板产品，更不是一个单点自动化产品。

它是一个用于管理数字化劳动力、服务真实运营结果的平台。

## 核心用户

OpsFactory 的主要使用者，不是组织里的每一位工程师，而是少量具备丰富经验的人类管理者。他们通过管理智能体来管理结果。

### FO Lead

FO Lead 负责面向服务交付的结果。

他关心的是：

- 工单生命周期
- SLA 达成情况
- 积压与超时
- 组别与团队负载
- 升级与流转模式
- 周报、月报与改进行动

他的主工作台是 **服务智能**。

### BO Lead

BO Lead 负责技术稳定性与运行保障结果。

他关心的是：

- 整体健康态势
- 告警质量与聚类情况
- 故障诊断
- 风险聚集点
- 稳定性退化趋势
- 技术治理优先级

他的主工作台是 **运维智能**。

### SDM

SDM 负责整体服务与稳定性结果。

他关心的是：

- 服务承诺是否兑现
- 系统是否稳定
- 智能体队伍是否高效产出
- 哪些问题需要管理介入

他的主工作台是 **首页**，即总控台。

## 产品定位

OpsFactory 的产品定位应长期保持稳定：

**OpsFactory 是一个 AI Native 的运维与服务运营平台，在这里，人类负责管理结果，智能体负责规模化完成运营工作。**

这个定义意味着：

- 平台的中心是管理与编排，而不只是交互
- 平台必须始终保留人类在环
- 平台必须贯通数据、洞察、行动与能力沉淀
- 平台必须能向第三方能力开放，而不是封闭应用

## 产品原则

以下原则应长期指导产品判断。

### 1. AI 是生产力，不是外挂

AI 不应只是挂在页面边上的助手。智能体本身就是平台的主生产力之一。

### 2. 人始终在指挥链上

OpsFactory 不是“完全无人运维”的幻想。平台服务的是经验丰富的人类 Lead，他们负责判断、审批、指挥与治理。

### 3. 洞察必须通向行动

没有行动的报表是不完整的，没有上下文的对话也是不完整的。OpsFactory 必须连接观测、推理、决策、执行与回看。

### 4. 每一次成功都应沉淀为能力

一次有效的分析，不应停留在单次会话里。它应有机会沉淀为 Prompt、Skill、Memory、MCP 或可复用的 Capability。

### 5. 智能体运营本身是产品表面，不是底层细节

如果 OpsFactory 的未来是 cyber factory，那么智能体的配置、执行、队列、产出与治理就必须是产品层面的核心能力，而不是隐藏在系统背后。

### 6. 平台必须开放

OpsFactory 必须能够消费外部能力，也必须能够向外开放能力。这包括第三方工具、企业内部知识、外部数据系统与未来生态伙伴。

## 产品范围

OpsFactory 应围绕三个核心业务域与一个统一入口来组织。

### 0. 首页

首页是总控台。

它不是欢迎页，也不是一个通用聊天入口页。它的职责是让人类 Lead 在最短时间内看清服务交付状态、技术运行状态与智能体执行状态。

### 1. 服务智能

服务智能是面向服务运营与流程治理的管理层。

它基于 ITSM 相关数据，回答这样的问题：

- 服务承诺是否被兑现
- 哪些工单在积压、超时、升级
- 哪些组别负载失衡
- 哪些流程环节拖慢了交付
- 本周、本月应输出怎样的管理摘要与改进行动

服务智能不是“报表中心”，而是：

**AI 驱动的服务交付运营台**

### 2. 运维智能

运维智能是面向系统运行与技术运营的管理层。

它基于 ITOM 相关数据，回答这样的问题：

- 当前整体健康态势如何
- 哪些告警是关键问题，哪些是噪音
- 哪些服务处于风险中
- 哪些异常在持续积累
- 现在最应该优先调查或缓解什么

运维智能不是“监控页面升级版”，而是：

**AI 驱动的技术运行决策台**

### 3. Ops 智能体

Ops 智能体是面向执行与生产的工作层。

它是任务被发起、执行、追踪、沉淀的地方，同时也包含为智能体执行服务的相关工作面。

Ops 智能体不是“若干助手的列表”，而是 cyber factory 的起点。

## 核心信息架构

OpsFactory 的核心信息架构应尽量简单、稳定，并与当前实现保持可演进的一致性。

```text
首页

服务智能
- 总览
- SLA 达成率
- 组别负载
- AI 周报 / 月报

运维智能
- 总览
- 健康态势
- 告警分析
- AI 洞察

Ops 智能体
- 发起任务
- 智能体目录
- 任务执行
- 收件箱
- 历史记录
- 文件
- 定时任务
- 产出物
- 能力配置
```

这里有一个重要取舍：

当前阶段不把“工作能力”单独拆成顶层域，而是统一归入 **Ops 智能体** 之下。

这是当前阶段更合理的选择，因为：

- 它让产品主骨架保持清晰
- 它强化了这些能力都是为智能体执行服务的认知
- 它与当前实现方向更接近，不会和现有页面结构产生过大冲突

结合当前代码实现，`chat`、`history`、`files`、`scheduled-actions`、`agents`、`agent configure` 更适合被统一理解为 Ops 智能体域的一部分，只是未来在用户可见导航与命名上可以继续演进。

## 核心交互架构

OpsFactory 的交互不应以“对话”为中心，而应以“管理闭环”为中心。

核心交互模型是：

```text
看见
-> 理解
-> 决策
-> 派发
-> 复核
-> 沉淀
```

把它放进产品里，就是：

```text
首页 / 服务智能 / 运维智能
-> 发现风险、异常、负载问题或优化机会
-> 下钻查看上下文与 AI 解读
-> 决定下一步动作
-> 把任务交给 Ops 智能体
-> 回看结果、报告与任务状态
-> 将有效经验沉淀为可复用能力
```

这也是 OpsFactory 和普通 AI 助手最根本的交互差异：

产品的中心不是“提问”，而是“管理”。

## 首页定义

首页必须是最重要的统一入口。

它要帮助 SDM、FO Lead、BO Lead 快速回答三个问题：

1. 服务交付是否健康
2. 技术运行是否健康
3. 哪些智能体工作需要关注或介入

因此首页的角色应同时是：

- SDM 的总控驾驶舱
- FO Lead / BO Lead 的每日开工台
- 三大业务域的统一分发入口

首页不能退化成“你好，我是某某助手”的欢迎页。

## Ops 智能体的角色

Ops 智能体值得被单独强调，因为它代表了 OpsFactory 与传统产品最大的差异化方向。

在当前阶段，Ops 智能体可能仍然更像一组实用工作面：

- 任务入口
- 会话执行
- 文件与产出管理
- 历史记录
- 定时执行
- 智能体配置

这在 MVP 阶段是可以接受的。

但产品意图必须提前明确：

**Ops 智能体是平台的生产操作系统。**

它未来应从：

- 会话入口
- 智能体浏览
- 文件与结果管理
- 定时执行

逐步演进为：

- 智能体劳动力管理
- 任务分发与排队
- 执行状态监控
- 产出治理
- 智能体班组
- 生产指标与人工介入面

即使 MVP 还做不到这一切，命名、导航与功能演进也应朝这个方向对齐。

## 开放能力平台

OpsFactory 必须从一开始就保留平台开放性。

当前产品已经具备正确的早期信号：gateway 统一入口、agent 配置、MCP、skills、prompts、memory，以及 capability market 的雏形。

这部分不应被视为零散功能，而应被定义为平台的开放能力层。

它至少包括：

- 通过 MCP 与 gateway 连接外部工具与系统
- 通过 Skill 沉淀标准作业知识
- 通过 Prompt 约束与塑造智能体行为
- 通过 Memory 保留组织上下文
- 通过 Capability Center / Market 封装与分发能力

这部分战略上非常重要，因为它决定 OpsFactory 最终是一个封闭应用，还是一个可生长的平台。

长期来看，OpsFactory 不仅要内部使用能力，还要成为：

- 能力被创建的地方
- 能力被治理的地方
- 能力被组装的地方
- 能力被开放给第三方的地方

## MVP 边界

MVP 不需要一次做成完整的 cyber factory，但必须从第一天开始说对产品语言。

MVP 最重要的是证明三件事：

1. 三大业务域的结构是清晰且有价值的
2. 首页是真正的总控台，而不是聊天首页
3. Ops 智能体被理解为执行域，而不只是助手列表

只要这三件事成立，后续无论扩展更深的 ITOM、ITSM 还是能力平台，都只是在加强同一个产品身份，而不是不断改写定义。

## 不可违背的产品约束

产品框架必须尊重当前实现现实。

- gateway 仍是 UI 与 SDK 的唯一稳定入口
- agent 的个性化行为应保持可配置，而不是写死在无关页面
- 能力开放应建立在现有 gateway、agent config、MCP、skill、prompt、memory 机制之上
- 新的信息架构与导航设计，应尽量减少与当前页面与路由骨架的冲突，并通过渐进演进完成转型

这些不只是工程约束，也是 OpsFactory 保持平台一致性的边界条件。

## 最终定义

如果把这份文档压缩成一句长期有效的定义，应当是：

**OpsFactory 是一个 AI Native 的人机协同运维平台，在这里，服务智能、运维智能与智能体执行被统一到一个可管理、可扩展、可开放的系统中。**

更通俗一点说：

**OpsFactory 帮助少数专家型人类管理大量智能体，以规模化地完成服务运营与技术运维工作。**
